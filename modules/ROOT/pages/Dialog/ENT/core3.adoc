== Overview

The Enterprise (ENT) AAA system focuses on core authentication, IP assignment, and subscriber provisioning using RADIUS. This section describes the essential functional components: Router-Based Authentication, Framed Address Handling, RADIUS Processing, and Provisioning Flow.

== RADIUS Process

The system uses RADIUS protocol for authentication and accounting between the router (NAS) and AAA server.

=== Authentication Flow

[source,unlang]
----
sequenceDiagram
    participant Router
    participant AAA
    participant DB

    Router->>AAA: Access-Request
    AAA->>DB: Validate User
    DB-->>AAA: Result
    AAA->>Router: Access-Accept / Reject
----

=== Accounting Flow

[source,unlang]
----
sequenceDiagram
    participant Router
    participant AAA

    Router->>AAA: Accounting-Start
    Router->>AAA: Accounting-Interim
    Router->>AAA: Accounting-Stop
----

=== Key Attributes

* User-Name
* User-Password
* NAS-IP-Address
* Framed-IP-Address
* Session-Timeout

== Router-Based Authentication

=== Flow Diagram

[source,unlang]
----
flowchart LR
    Router -->|Access-Request| AAA
    AAA --> DB
    AAA -->|Accept/Reject| Router
----

=== Decision Logic

[source,unlang]
----
flowchart TD
    A[Request] --> B{User Exists?}
    B -- No --> X[Reject]
    B -- Yes --> C{Password Valid?}
    C -- No --> X
    C -- Yes --> D{Account Active?}
    D -- No --> X
    D -- Yes --> E[Accept]
----

=== Attribute-Level Details

| Attribute | Description |
|----------|------------|
| Framed-IP-Address | Assigns user IP |
| VRF-Name | Routing context |
| Session-Timeout | Session duration |

=== Failure Scenarios

* Invalid credentials → Reject
* User not found → Reject
* Account inactive → Reject
* AAA timeout → Retry/Fail

== Framed Address Handling

=== Flow Diagram

[source,unlang]
----
flowchart TD
    A[Check IP] --> B{Request IP?}
    B -- Yes --> R[Use Request]
    B -- No --> C{DB IP?}
    C -- Yes --> D[Use DB]
    C -- No --> E{External IP?}
    E -- Yes --> F[Use External]
    E -- No --> G[Reject]
----

=== Decision Logic

* Request IP takes priority
* DB IP used if request absent
* External IP used as fallback
* Reject if none available

=== Attribute-Level Details

| Attribute | Description |
|----------|------------|
| Framed-IP-Address | Assigned IP |
| Framed-Pool | Pool reference |
| Framed-IP-Netmask | Subnet |

=== Failure Scenarios

* No IP available → Reject
* Pool exhausted → Reject
* Duplicate IP → Conflict

== Session Lifecycle

The Enterprise AAA system implicitly follows a session lifecycle through RADIUS authentication and accounting processes.

=== Lifecycle Stages

. Authentication → Access-Request / Access-Accept
. Session Start → Accounting-Start
. Active Session → Interim Updates
. Session Termination → Accounting-Stop

=== Example Flow

[source,unlang]
----
sequenceDiagram
    participant Router
    participant AAA
    participant DB

    Router->>AAA: Access-Request
    AAA->>DB: Validate User
    AAA->>Router: Access-Accept
    Router->>AAA: Accounting-Start
    loop Active Session
        Router->>AAA: Interim-Update
    end
    Router->>AAA: Accounting-Stop
----

=== Example SQL Representation

Session data can be stored and tracked using a table structure like below:

[source,sql]
----
CREATE TABLE user_sessions (
    session_id VARCHAR(64) PRIMARY KEY,
    username VARCHAR(64),
    framed_ip VARCHAR(45),
    start_time TIMESTAMP,
    last_update TIMESTAMP,
    stop_time TIMESTAMP,
    input_octets BIGINT,
    output_octets BIGINT,
    status VARCHAR(20)
);

-- Insert on Accounting-Start
INSERT INTO user_sessions (session_id, username, framed_ip, start_time, status)
VALUES ('ABC123', 'user1', '10.0.0.10', NOW(), 'ACTIVE');

-- Update on Interim-Update
UPDATE user_sessions
SET last_update = NOW(),
    input_octets = input_octets + 5000,
    output_octets = output_octets + 3000
WHERE session_id = 'ABC123';

-- Update on Accounting-Stop
UPDATE user_sessions
SET stop_time = NOW(),
    status = 'TERMINATED'
WHERE session_id = 'ABC123';
----

=== Purpose

* Tracks active user sessions in real-time
* Maintains usage data for billing and monitoring
* Ensures session continuity even if stop packets are delayed
