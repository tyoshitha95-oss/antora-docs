= Overview

The GPON User Management API provides a comprehensive interface to manage GPON subscribers. The API enables:

* Creation of new GPON users
* Package updates
* Password updates
* Barring and service activation/deactivation

All operations are **performed using the HTTP PUT method**, making them idempotent and suitable for integration with batch processing or real-time provisioning systems. Each API request requires **Basic HTTP Authentication**.

**Base URLs**

* Staging environment: `http://10.62.167.228:80`
* Production environment: `http://10.62.167.150:8080`

**Core Capabilities:**

1. **Create Subscriber** – Supports Public Static, Private Dynamic, and Public Dynamic IP assignments.
2. **Update Package** – Allows changing the subscriber’s service package without altering other user attributes.
3. **Update Password** – Modify the ONT/ONU device password securely.
4. **Change Barring Status** – Enable or disable service at any time with immediate effect.
5. **Real-Time Monitoring** – Provides user status fields such as `status`, `barringstatus`, `last_update`.

**Request Format:**  
All payloads are `application/x-www-form-urlencoded`. Parameter names must match exactly, including `customfield11`, `customfield10`, `customfield12`, `customnumfield1`, etc.

**Response Format:**  
All API responses return HTTP status `200` for successful operations.  

[source,json]
----
{
  "status": "SUCCESS",
  "message": "GPON user updated successfully",
  "statusCode": 200,
  "data": { ...full user object... }
}
----

**Important Notes:**

* All requests must include valid authentication credentials.
* The APIs are idempotent; repeated requests with the same data will not create duplicate entries.
* Certain parameters are **mandatory** depending on the chosen IP assignment type.
* All response fields are provided for auditing and troubleshooting, including `create_user`, `create_date`, `last_update`, and `barringstatus`.
* Clients should always check the `status` field before processing the `data` object.
* Use logging for both request and response to maintain audit trails.
* IPv4 and IPv6 assignment parameters differ by use case and must be handled accordingly.
* API clients should implement retries with exponential backoff for transient network failures.
* Session and idle timeout parameters (`maxusersessions`, `sessionlimit`, `customnumfield1`) must be configured carefully to prevent unnecessary disconnects.
* IP pools for dynamic assignment are defined by `customfield13` for public dynamic users.
* Static assignments require both `userip` and `framedIPv6prefix`.
* Ensure that `groupname` is always `GPON` to match RADIUS configurations.
