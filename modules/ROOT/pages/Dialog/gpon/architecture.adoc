= System Architecture

The GPON User Management API serves as the **central provisioning layer** for managing GPON subscribers. It interfaces between upstream business systems (billing, CRM, provisioning platforms) and the downstream GPON infrastructure (OLT/ONT, RADIUS, PCRF).  

The architecture is designed to **support high-volume, real-time provisioning**, while maintaining session control, IP allocation, and audit compliance.

=== Components Overview

* **Frontend Layer**  
  - External systems (e.g., billing, CRM, OSS/BSS) communicate with the API over HTTP.  
  - Requests include subscriber information, package selection, and authentication credentials.  
  - Supports both single-user and bulk operations.

* **Authentication Layer**  
  - All requests are authenticated using **Basic HTTP Authentication**.  
  - Credentials are validated against an internal directory.  
  - Unauthorized requests receive HTTP 401 responses.

* **Business Logic Layer**  
  - Validates all request parameters (user ID, package, session limits, IP assignment type).  
  - Determines the correct IP assignment strategy:
    ** Public Static
    ** Private Dynamic
    ** Public Dynamic  
  - Populates RADIUS attributes, including `userip`, `framedIPv6prefix`, `Framed-Pool`, `maxusersessions`, and `sessionlimit`.  
  - Handles idempotency: repeated requests update existing subscriber records without creating duplicates.

* **Backend Integration Layer**  
  - **PCRF Integration**: `customfield11` links the subscriber to the Policy and Charging Rules Function.  
  - **RADIUS Attribute Population**: Assigns session parameters, IP information, and group membership.  
  - **OLT/ONT Provisioning**: Commands sent downstream to provision subscriber devices.  
  - Ensures atomic updates and real-time policy application.

* **Data Store**  
  - Persistent user records with all attributes (`user_id`, `password`, `customfield10`, `customfield12`, `userip`, `framedIPv6prefix`, `customfield13`, `barringstatus`).  
  - Audit fields: `create_user`, `create_date`, `last_update` for traceability.  
  - Stores session parameters and idle timeout information for monitoring.

* **Deployment**  
  - Operates on an internal private network (RFC 1918 addresses).  
  - **Staging Environment**: `10.62.167.228:80` — used for testing and validation.  
  - **Production Environment**: `10.62.167.150:8080` — used for live provisioning.  
  - Environments are isolated to prevent accidental cross-deployment updates.

=== Data Flow

1. Client sends HTTP PUT request with subscriber data.  
2. Authentication layer validates credentials.  
3. Business logic checks IP assignment type and package parameters.  
4. RADIUS attributes are populated based on IP, session, and package values.  
5. Changes are applied to backend databases and OLT/ONT devices.  
6. JSON response is returned with the updated user object.  
7. Audit logs are generated with `create_user`, `create_date`, and `last_update`.

=== IP Assignment Strategies

* **Public Static**  
  - Both IPv4 (`userip`) and IPv6 (`framedIPv6prefix`) are explicitly assigned.  
  - Ensures consistent connectivity for business users requiring fixed addresses.

* **Private Dynamic**  
  - IPv4 is allocated dynamically from internal pools.  
  - IPv6 may be optionally delegated.  
  - Suitable for residential deployments.

* **Public Dynamic**  
  - IP assigned from a named pool (`customfield13`).  
  - No static IP required; ensures public addressing without manual assignment.  

=== Session Management

* `maxusersessions`: Maximum simultaneous sessions per subscriber.  
* `sessionlimit`: Maximum session duration in seconds.  
* `customnumfield1`: Idle timeout; automatically disconnects inactive sessions.  
* The API enforces session limits atomically to prevent table overload in RADIUS servers.

=== Audit & Monitoring

* All changes are logged with `create_user` and timestamps.  
* `last_update` allows administrators to detect stale or incomplete updates.  
* Session activity and barring status can be monitored via API responses.

=== Deployment Considerations

* High-volume provisioning requires careful tuning of session and idle timeout parameters.  
* For Public Dynamic IP pools, ensure sufficient addresses to avoid allocation conflicts.  
* Staging environment should mirror production for testing before deployment.  
* TLS termination is recommended at reverse proxy for secure transport.  
* Logging and monitoring are essential for identifying failures or misconfigurations.  

=== Summary

The architecture ensures:

* **Atomic provisioning** — subscriber updates are consistent and idempotent.  
* **Session enforcement** — prevents simultaneous session overload.  
* **Real-time IP assignment** — supports static and dynamic IP strategies.  
* **Auditability** — all operations are traceable via logs and timestamps.  
* **Scalability** — designed for large-scale subscriber bases with dynamic or static IPs.  

This architecture provides a **reliable, secure, and auditable platform** for managing GPON subscribers in real-time across staging and production environments.
