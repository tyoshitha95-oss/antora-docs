= Error Handling

This page details expected API behavior and error management.

The GPON User Management API is designed with **idempotency** in mind. Most operations return `SUCCESS` even if repeated, as long as the final state matches the requested payload.

**Standard Response Fields**

[source,json]
----
{
  "status": "SUCCESS" | "FAILURE",
  "message": "Descriptive message",
  "statusCode": 200 | 400 | 401 | 500,
  "data": { ...user object... }
}
----

**Common Error Scenarios**

1. **Authentication Failure**  
   * Cause: Invalid or missing Basic Auth credentials.  
   * Response: HTTP 401 Unauthorized.  
   * Recommendation: Verify credentials and retry.

2. **Parameter Validation Error**  
   * Cause: Missing required fields (e.g., `customfield11`, `password`) or invalid formats.  
   * Response: HTTP 400 Bad Request.  
   * Recommendation: Validate all required parameters prior to API submission.

3. **Duplicate User Creation**  
   * Cause: User already exists with same `user_id`.  
   * Response: `SUCCESS` (idempotent design).  
   * Note: Existing user details are updated; no duplicate record is created.

4. **Invalid IP Assignment**  
   * Cause: Specifying `userip` or `framedIPv6prefix` incorrectly.  
   * Response: May return `SUCCESS` but result in backend mismatch.  
   * Recommendation: Always verify IP parameters per assignment type.

5. **System Errors**  
   * Cause: Backend service or RADIUS server failure.  
   * Response: HTTP 500 Internal Server Error.  
   * Recommendation: Implement retry logic and alert operations team.

**Operational Guidelines**

* Always check `status` field in JSON response before processing `data`.  
* Use `message` field for diagnostics and logging.  
* Maintain correlation between request timestamp and `create_date`/`last_update` for auditing.  
* Ensure all automated scripts handle transient failures gracefully using retries with exponential backoff.  
* Validate responses against the expected IP assignment and package values.

**Extended Notes**

* The API does not document all failure codes exhaustively. Network-level errors (timeouts, connection failures) are handled externally.  
* Bulk operations should log each request/response to detect partial failures.  
* Monitoring `last_update` can help identify stuck or stale records in automated provisioning pipelines.  
* Implement alerts for repeated failures to catch misconfigured parameters early.
