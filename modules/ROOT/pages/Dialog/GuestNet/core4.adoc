= Core Function Analysis
:toc: left
:toclevels: 4
:sectnums:

The GuestNet system processes all RADIUS requests through a central handler function called `handle_request`. This function acts as the main decision engine and routes requests based on their type, ensuring proper handling of authentication and accounting flows.

== 3.1 Request Routing Logic

The system classifies incoming RADIUS packets and directs them to the appropriate function.

[mermaid]
----
flowchart TD
    A[RADIUS Packet Received] --> B{Packet Type}
    B -->|Access-Request| C[AuthUser]
    B -->|Acct-Start| D[AcctStart]
    B -->|Acct-Alive| E[AcctAlive]
    B -->|Acct-Stop| F[AcctStop]
    B -->|Other| G[Accept Default]
----

* Access-Request → `AuthUser` (Authentication)
* Accounting-Start → `AcctStart`
* Accounting-Alive → `AcctAlive`
* Accounting-Stop → `AcctStop`

---

== 3.2 Authentication Flow (AuthUser)

The `AuthUser` function is responsible for validating user access and assigning session parameters.

=== 3.2.1 Authentication Workflow

[mermaid]
----
flowchart TD
    A[Access-Request] --> B[Load Cache]
    B --> C[Extract Credentials]
    C --> D[Resolve Location]
    D --> E[Query ACCOUNT]
    E --> F{User Exists?}
    F -->|No| F1[Reject]
    F -->|Yes| G{Password Valid?}
    G -->|No| G1[Reject]
    G -->|Yes| H{Package Allowed?}
    H -->|No| H1[Reject]
    H -->|Yes| I{Concurrency OK?}
    I -->|No| I1[Reject]
    I -->|Yes| J{Status Active?}
    J -->|No| J1[Reject]
    J -->|Yes| K{Expired?}
    K -->|Yes| K1[Reject]
    K -->|No| L{Quota Available?}
    L -->|No| L1[Reject]
    L -->|Yes| M[Calculate Session]
    M --> N[Build Response]
    N --> O[Access-Accept]
----

---

=== 3.2.2 Cache Loading

To improve performance, the system loads frequently used data into memory.

[source,sql]
----
SELECT ID, CI FROM LOCATION WHERE STATUS=0;
SELECT ID, TIME_QUOTA, VALID_FOR FROM PACKAGES WHERE STATUS=0;
----

Purpose:
- Reduce database load
- Enable fast validation of location and package mappings

---

=== 3.2.3 User Validation

[source,sql]
----
SELECT 
    unix_timestamp(CREATE_DATE) AS CD,
    IFNULL(unix_timestamp(FIRST_LOGIN),0) AS FL,
    unix_timestamp(NOW()) AS CT,
    PASSWORD,
    PKG_ID,
    LOC_ID,
    USED_TIME_QUOTA,
    STATUS
FROM ACCOUNT
WHERE PIN = '<username>'
LIMIT 1;
----

Purpose:
- Validate user existence
- Retrieve quota, package, and status

---

=== 3.2.4 Password Validation

Two modes are supported:

- Normal password validation
- Master password (`dialog`) bypass

Decision:
- If password mismatch → Reject
- Else → Continue

---

=== 3.2.5 Location Resolution

The system determines the user location using two methods:

==== L2 Detection (Circuit-ID)
[source,sql]
----
SELECT CI
FROM WIFI.WIFI_IP
WHERE circuit_id = '<circuit_id>'
AND remote_id = '<remote_id>'
LIMIT 1;
----

==== L3 Detection (IP-Based)
[source,sql]
----
SELECT CI
FROM WIFI.WIFI_IP
WHERE start_ip <= inet_aton('<ip>')
AND inet_aton('<ip>') <= end_ip
LIMIT 1;
----

---

=== 3.2.6 Package Validation

[source,sql]
----
SELECT 1
FROM PKG_TO_LOC
WHERE PKG_ID = '<pkg_id>'
AND LOC_ID = '<loc_id>';
----

Purpose:
- Ensure user package is allowed at the current location

---

=== 3.2.7 Concurrency Control

[source,sql]
----
SELECT COUNT(*) AS CNT
FROM WIFI.RADONLINE
WHERE USERNAME LIKE '<username>%';
----

Purpose:
- Enforce maximum allowed simultaneous sessions

---

=== 3.2.8 Session Timeout Calculation

Session timeout depends on package type:

[cols="1,2,2"]
|===
|PKGTYPE |Expiry Base |Session Logic

|1 |First Login |Until expiry
|2 |Create Date |Until expiry
|3 |First Login |Quota-based
|4 |Create Date |Quota-based
|===

Rules:
- Reject if expired
- Reject if quota exhausted
- Maximum session capped at 7 days

---

=== 3.2.9 RADIUS Response Attributes

If authentication succeeds:

* Session-Timeout
* Idle-Timeout
* Class attribute
* Service-Info
* Account-Info
* Vendor-specific attributes

---

== 3.3 Accounting Flow

Accounting tracks session lifecycle and usage.

---

=== 3.3.1 Accounting Start (AcctStart)

[mermaid]
----
flowchart TD
    A[Acct-Start] --> B[Extract Attributes]
    B --> C[Parse Class]
    C --> D[Delete Old Session]
    D --> E[Insert RADONLINE]
    E --> F[Set First Login]
    F --> G[Accept]
----

==== SQL Operations

[source,sql]
----
DELETE FROM WIFI.RADONLINE WHERE FRAMEDIPADDRESS = '<ip>';
----

[source,sql]
----
INSERT INTO WIFI.RADONLINE (...)
VALUES (...);
----

[source,sql]
----
UPDATE ACCOUNT
SET FIRST_LOGIN = NOW()
WHERE FIRST_LOGIN IS NULL
AND PIN = '<username>';
----

---

=== 3.3.2 Accounting Interim (AcctAlive)

[mermaid]
----
flowchart TD
    A[Acct-Alive] --> B[Fetch Previous Session]
    B --> C[Calculate Delta]
    C --> D[Update Quota]
    D --> E[Update RADONLINE]
    E --> F[Accept]
----

==== SQL Operations

[source,sql]
----
SELECT sessionTime, download, upload
FROM WIFI.RADONLINE
WHERE ACCTSESSIONID = '<session_id>';
----

[source,sql]
----
UPDATE ACCOUNT
SET USED_TIME_QUOTA = USED_TIME_QUOTA + <delta_time>
WHERE PIN = '<username>';
----

---

=== 3.3.3 Accounting Stop (AcctStop)

[mermaid]
----
flowchart TD
    A[Acct-Stop] --> B[Fetch Session]
    B --> C[Calculate Final Delta]
    C --> D[Update Quota]
    D --> E[Delete RADONLINE]
    E --> F[Insert CDR]
    F --> G[Insert CDRS]
    G --> H[Accept]
----

==== SQL Operations

[source,sql]
----
DELETE FROM WIFI.RADONLINE
WHERE ACCTSESSIONID = '<session_id>';
----

[source,sql]
----
INSERT INTO CDR (...)
VALUES (...);
----

[source,sql]
----
INSERT INTO WIFI.CDRS (...)
VALUES (...);
----

== 3.4 Class Attribute Lifecycle

The `Class` attribute carries session metadata across all stages.

Format:
----
LOCID:<loc>|CUID:<user>|PKG:<pkg>|CI:walkin|GCUID:<id>
----

Usage:
- Authentication → Created
- Accounting Start → Parsed
- Interim Updates → Maintained
- Stop → Stored in CDR

---

== 3.6 Key Design Principles

- Cache-driven validation for performance
- Delta-based accounting for accuracy
- Stateless session tracking via Class attribute
- Strong separation of authentication and accounting
- Optimized database interaction

---

== 3.7 Summary

The Core Function Analysis demonstrates that GUESTNET:

- Provides robust RADIUS-based authentication
- Ensures strict policy enforcement
- Tracks sessions accurately using accounting flows
- Maintains high performance via caching
- Supports scalable and reliable guest WiFi access
